# Design: client_bot multi-master scenarios

**Date:** 2026-03-31
**Scope:** Минимальные правки в двух функциях — `get_client_context` и `cmd_start`.
**Not in scope:** FSM state, master_bot, мастерский Mini App, регистрационный FSM.

---

## Проблема

Из предыдущей сессии реализованы: `_active_masters` dict, `show_master_select`, кнопка "Сменить мастера", callback-обработчики. Не реализованы три сценария:

| Сценарий | Текущее поведение | Нужное |
|----------|-----------------|--------|
| Существующий клиент + токен нового мастера | Токен игнорируется, показывается Home | Привязать → приветствие → Home |
| Существующий клиент + токен уже привязанного мастера | Home без сообщения | "Вы уже записаны к {имя}" → Home |
| `/start` без токена, 2+ мастеров | `show_master_select` (выбор экран) | Home с best_master + кнопка "Сменить мастера" |

---

## Решение

### 1. `get_client_context` — fallback на latest last_visit

**Изменение:** строка 175-177. Вместо `return client, None, None` при 2+ мастерах — взять `masters[0]` (уже отсортированы по `last_visit DESC NULLS LAST` в SQL-запросе).

```python
else:
    master_id = masters[0]["master_id"]  # latest last_visit
```

Это исправляет сценарий 5: `cmd_start` без токена теперь всегда получает валидный контекст для мультимастерного клиента.

### 2. `cmd_start` — переставить порядок проверок

**Текущий порядок:**
1. `get_client_context` → если есть мастер → Home (токен игнорируется!)
2. если `client and not master` → `show_master_select`
3. извлечь токен

**Новый порядок:**
1. Загрузить `client` (без master)
2. Извлечь токен из аргументов
3. **Если токен есть:**
   - Найти мастера по токену
   - Если клиент не существует → полная регистрация (FSM, как раньше)
   - Если клиент уже привязан к этому мастеру → "Вы уже записаны к {имя}" + Home
   - Если клиент существует, но не привязан → `link_client_to_master` + `accrue_welcome_bonus` + "✅ Вы подключились к {имя}!" + Home
4. **Если токена нет:**
   - Клиента нет → "Нужна ссылка от мастера"
   - Клиент есть → `get_client_context` → Home (fallback на last_visit работает)

### 3. Хранение active_master после сценария 2

После привязки к новому мастеру записать в `_active_masters[tg_id] = master.id`, чтобы Home отобразился для нового мастера.

## Критерии готовности

1. `/start {token}` нового клиента → регистрация как раньше ✓
2. `/start {token}` существующего, новый мастер → привязка + "Подключились к {имя}"
3. `/start {token}` существующего, уже привязан → "Уже записаны к {имя}" + Home
4. `/start` без токена, 1 мастер → Home как раньше ✓
5. `/start` без токена, 2+ мастеров → Home + кнопка "Сменить мастера"
6. `python3 -c "from src.client_bot import router; print('OK')"` → OK
